# Drools 订单积分项目代码文档

> 项目：`drools_project`
>
> 技术栈：Java 17、Spring Boot 3.0.5、Drools 8.41.0.Final、JUnit 5
> 文档范围：当前仓库中的订单积分示例，覆盖依赖、规则编译、会话执行、启动演示和测试用例。

## 1. 项目目的与处理链路

本项目使用 Drools 为订单金额计算积分。Java 代码只负责提供事实数据（`Order`）与触发规则；金额区间和积分值保存在 DRL 规则文件中，因此后续调整积分策略时通常无需改动业务流程代码。

完整调用链如下：

```text
pom.xml 引入 Drools 依赖
        |
        v
DroolsConfig 读取 classpath:rules/order.drl
        |
        v
KieFileSystem -> KieBuilder.buildAll() -> KieModule -> KieContainer（Spring Bean）
        |
        v
应用启动器或测试类注入 KieContainer
        |
        v
创建 KieSession，插入 Order 事实
        |
        v
fireAllRules() 匹配 DRL 的 when 条件并执行 then 动作
        |
        v
Order.score 被规则写入，dispose() 释放会话
```

项目目录说明：

```text
drools_project/
├── pom.xml                                      Maven 依赖与版本配置
├── src/main/java/com/drools/
│   ├── DroolsApplication.java                   Spring Boot 入口和启动演示
│   ├── config/DroolsConfig.java                 KieContainer Bean 配置
│   └── model/Order.java                         规则事实（Fact）模型
├── src/main/resources/rules/order.drl           订单积分规则
└── src/test/java/com/drools/
    └── DroolsDemosApplicationTests.java         Spring Boot 集成测试
```

## 2. 环境与依赖引入

- JDK：`pom.xml` 指定编译目标为 Java 17；请使用 JDK 17 或更高版本执行 Maven。
- Maven：项目使用 Maven 管理依赖并执行测试。
- 网络或本地 Maven 缓存：首次执行 Maven 时需要下载 Spring Boot 与 Drools 依赖。

常用命令：

```bash
cd drools_project
mvn test
mvn spring-boot:run
```

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
```

3. 当前声明 `maven.compiler.source=17`，没有显式声明 `maven.compiler.target` 或 `maven.compiler.release`。如需严格保证字节码兼容 Java 17，建议后续改为 `<maven.compiler.release>17</maven.compiler.release>`。

## 3. 事实模型：Order

文件：`src/main/java/com/drools/model/Order.java`

```java
public class Order {

    private double amout;
    private double score;

    public double getAmout() {
        return amout;
    }

    public void setAmout(double amout) {
        this.amout = amout;
    }

    public double getScore() {
        return score;
    }

    public void setScore(double score) {
        this.score = score;
    }
}
```

`Order` 是插入 Drools 工作内存的事实对象（Fact）。规则通过 JavaBean 属性访问约定解析字段：DRL 中的 `amout` 实际会调用 `getAmout()`；规则动作中的 `$order.setScore(...)` 则直接调用 Java Setter。

字段含义：

| 字段 | 类型 | 用途 | 规则中的访问方式 |
| --- | --- | --- | --- |
| `amout` | `double` | 订单金额，用于匹配金额区间。 | `Order(amout >= 1000)` |
| `score` | `double` | 规则执行后的积分结果。 | `$order.setScore(1000)` |

**注意事项：**

1. `amout` 是 `amount` 的拼写错误，但 Java 属性名、Getter/Setter 和 DRL 条件目前保持一致，所以当前代码可以正常执行。重命名时必须同步修改 `Order.java`、`order.drl`、应用代码和测试代码；只改 Java 字段会导致规则解析或匹配失败。
2. 金额不建议在生产场景中使用 `double`。二进制浮点数可能造成金额比较精度问题，应优先使用 `BigDecimal`，并在 DRL 中使用 `compareTo` 或封装后的比较字段。
3. 若规则的 `then` 部分修改已插入事实中、并且还希望基于新值继续触发其他规则，需要通过 `modify($order) { setScore(...) }` 或 Java 侧 `session.update(factHandle, order)` 通知引擎。本示例仅计算一次积分，且没有依赖 `score` 的后续规则，因此当前写法能满足演示目的。

## 4. 规则文件：order.drl

文件：`src/main/resources/rules/order.drl`

### 4.1 DRL 文件头

```drl
//订单积分规则
package com.order
import com.drools.model.Order
```

- `package com.order` 是 DRL 规则包名，与 Java 包名可以不同。
- `import com.drools.model.Order` 使规则可直接写 `Order(...)`，而不必写全限定类名。
- 此文件位于 `src/main/resources`，Maven 会将其复制到运行时 classpath；配置类按 `rules/order.drl` 加载它。

### 4.2 规则语法与执行含义

每个规则由 `rule`、`when`、`then`、`end` 组成：

```drl
rule "规则名称"
    when
        $order : Order(条件表达式)
    then
        // 规则命中后执行的 Java 代码
end
```

- `rule "..."`：规则名称，在同一个 KIE 模块中应保持唯一，便于日志、审计和排错。
- `when`：左侧条件（LHS）。`$order : Order(...)` 会从工作内存中找出类型为 `Order` 且满足条件的对象，并绑定到变量 `$order`。
- `then`：右侧动作（RHS）。可调用匹配事实的方法，例如 `$order.setScore(1000)`。
- `end`：结束当前规则定义。

### 4.3 当前积分规则逐条解析

```drl
rule "order_rule_1"
    when
        $order:Order(amout < 100)
    then
        $order.setScore(0);
        System.out.println("成功匹配到规则一：100元以下 不加分");
end
```

金额小于 100 时积分为 0。由于条件严格小于 `100`，金额恰好等于 100 不会命中本规则，而会命中下一条规则。

```drl
rule "order_rule_2"
    when
        $order:Order(amout >= 100 && amout < 500)
    then
        $order.setScore(100);
        System.out.println("成功匹配到规则二：100元 - 500元 加100分");
end
```

金额范围为 `[100, 500)`，即 100 到 499.999...，积分为 100。

```drl
rule "order_rule_3"
    when
        $order:Order(amout >= 500 && amout < 1000)
    then
         $order.setScore(500);
         System.out.println("成功匹配到规则三：500元 - 1000元 加500分");
end
```

金额范围为 `[500, 1000)`，积分为 500。

```drl
rule "order_rule_4"
    when
        $order:Order(amout >= 1000)
    then
         $order.setScore(1000);
         System.out.println("成功匹配到规则四：1000元以上 加1000分");
end
```

金额范围为 `[1000, +infinity)`，积分为 1000。当前测试传入金额 1500，因此只会命中此规则。

### 4.4 区间总览

| 订单金额 | 命中规则 | 积分 |
| --- | --- | --- |
| `< 100` | `order_rule_1` | 0 |
| `>= 100` 且 `< 500` | `order_rule_2` | 100 |
| `>= 500` 且 `< 1000` | `order_rule_3` | 500 |
| `>= 1000` | `order_rule_4` | 1000 |

这些条件互斥且完整覆盖所有普通数值金额，所以每个 `Order` 事实在当前规则集下只会命中一条规则。

**注意事项：**

1. 边界的 `>=` 与 `<` 是规则正确性的关键。修改区间时要检查 100、500、1000 等临界值，避免产生重叠（同一订单执行多条规则）或空档（订单不执行任何规则）。
2. `System.out.println` 适合演示，不适合生产日志。实际项目宜使用日志框架，并避免在规则中写入敏感订单数据。
3. 规则的执行顺序不能只依赖文件书写顺序。当前条件互斥，所以顺序不影响结果；若将来条件可能同时满足，应使用 `salience`、`agenda-group`、`activation-group` 等机制明确控制策略。
4. 金额为负数时会命中第一条规则并得到 0 分。若业务上禁止负金额，应在进入规则引擎前校验，或增加拒绝/异常处理规则。

## 5. KIE 容器配置与规则编译

文件：`src/main/java/com/drools/config/DroolsConfig.java`

### 5.1 配置类和规则路径

```java
@Configuration
public class DroolsConfig {

    private static final KieServices kieServices = KieServices.Factory.get();
    private static final String RULES_CUSTOMER_RULES_DRL = "rules/order.drl";
```

`@Configuration` 表示该类用于注册 Spring Bean。`KieServices` 是 Drools/KIE 的入口工厂，负责创建文件系统、构建器、模块和容器。规则路径相对于 classpath，因此它与资源文件 `src/main/resources/rules/order.drl` 对应。

### 5.2 构建 KieContainer 的步骤

```java
@Bean
public KieContainer kieContainer() {
    KieFileSystem kieFileSystem = kieServices.newKieFileSystem();
    kieFileSystem.write(ResourceFactory.newClassPathResource(RULES_CUSTOMER_RULES_DRL));

    KieBuilder kieBuilder = kieServices.newKieBuilder(kieFileSystem);
    kieBuilder.buildAll();

    KieModule kieModule = kieBuilder.getKieModule();
    KieContainer kieContainer = kieServices.newKieContainer(kieModule.getReleaseId());

    return kieContainer;
}
```

执行顺序与各对象职责如下：

1. `newKieFileSystem()`：创建内存中的 KIE 文件系统。
2. `newClassPathResource("rules/order.drl")`：从应用 classpath 找到 DRL 文件。
3. `kieFileSystem.write(...)`：将规则资源写入待构建的 KIE 文件系统。
4. `newKieBuilder(...)`：创建规则构建器。
5. `buildAll()`：解析、编译全部规则并构建 KIE 模块。
6. `getKieModule()`：取得构建结果及其 `ReleaseId`。
7. `newKieContainer(...)`：基于构建出的模块创建可复用的 `KieContainer`。
8. `@Bean` 返回值：Spring 将该容器托管为单例 Bean，其他组件可通过依赖注入获取。

**注意事项：**

1. 当前实现没有检查 `kieBuilder.getResults()`。DRL 语法错误、类导入错误或资源路径错误时，建议在创建 `KieModule` 前检测错误并抛出异常，否则问题可能在运行时以不直观的形式出现。可采用：

```java
Results results = kieBuilder.getResults();
if (results.hasMessages(Message.Level.ERROR)) {
    throw new IllegalStateException(results.getMessages().toString());
}
```

使用该片段时需额外导入 `org.kie.api.builder.Results` 和 `org.kie.api.builder.Message`。

2. `rules/order.drl` 必须在最终运行包的 classpath 中。若更改资源目录、文件名或路径常量而不同步修改，`newClassPathResource` 将无法正确读取规则。
3. `KieContainer` 可以作为单例长期复用；`KieSession` 是有状态对象，不应作为 Spring 单例共享给并发请求。
4. 本项目未使用 `kmodule.xml`，而是通过编程方式写入并构建规则。因此 `newKieSession()` 使用默认会话配置。规则数量增多、需要多个规则库或明确命名会话时，可再引入 `META-INF/kmodule.xml` 管理。

## 6. 应用启动演示

文件：`src/main/java/com/drools/DroolsApplication.java`

```java
@SpringBootApplication
public class DroolsApplication {

    public static void main(String[] args) {
        SpringApplication.run(DroolsApplication.class, args);
    }

    @Bean
    public CommandLineRunner demo(KieContainer kieContainer) {
        return args -> {
            KieSession session = kieContainer.newKieSession();
            Order order = new Order();
            order.setAmout(1500);
            session.insert(order);
            session.fireAllRules();
            session.dispose();
            System.out.println("订单金额：" + order.getAmout() + "，增加积分：" + order.getScore());
        };
    }
}
```

代码解析：

1. `@SpringBootApplication` 开启组件扫描和自动配置，因此 `DroolsConfig` 中的 `KieContainer` Bean 会被发现。
2. `SpringApplication.run(...)` 完成 Spring 容器启动。
3. `CommandLineRunner` 是一个启动完成后执行一次的回调。Spring 根据方法参数自动注入 `KieContainer`。
4. `newKieSession()` 创建一份独立的有状态规则会话。
5. `order.setAmout(1500)` 设置事实数据。
6. `session.insert(order)` 将对象放入 Drools 工作内存；此时只是插入，还未执行规则。
7. `session.fireAllRules()` 评估议程中的激活规则。1500 满足 `order_rule_4`，规则把 `score` 设置为 1000。
8. `session.dispose()` 释放会话持有的资源。
9. 最终打印预期为：`订单金额：1500.0，增加积分：1000.0`。

**注意事项：**

1. `CommandLineRunner` 在每次 Spring Boot 应用启动时都会执行。若以后改为 Web 服务或生产任务，不能把真实订单处理逻辑直接放在启动回调里，否则应用重启会重复处理数据。
2. 应使用 `try/finally` 确保异常时也会调用 `dispose()`。推荐结构：

```java
KieSession session = kieContainer.newKieSession();
try {
    session.insert(order);
    session.fireAllRules();
} finally {
    session.dispose();
}
```

3. `fireAllRules()` 返回实际触发的规则数。需要诊断或测试时可保存该值，例如 `int fired = session.fireAllRules();`。

## 7. 测试用例解析

文件：`src/test/java/com/drools/DroolsDemosApplicationTests.java`

### 7.1 测试代码

```java
@SpringBootTest
public class DroolsDemosApplicationTests {

    @Autowired
    private KieContainer kieContainer;

    @Test
    public void test() {
        KieSession session = kieContainer.newKieSession();

        Order order = new Order();
        order.setAmout(1500);
```

### 7.2 测试执行过程

1. `@SpringBootTest` 启动完整 Spring Boot 测试上下文。
2. 测试上下文扫描到 `DroolsConfig`，执行 `kieContainer()`，编译并注册规则容器。
3. `@Autowired` 将该 `KieContainer` 注入测试类。
4. 测试创建独立 `KieSession`，构造金额为 1500 的 `Order`，插入后调用 `fireAllRules()`。
5. `order_rule_4` 被触发，将 `order.score` 设为 1000。
6. 会话释放后打印订单金额和积分。

已有 Surefire 测试报告显示：`Tests run: 1, Failures: 0, Errors: 0, Skipped: 0`，说明当前测试可正常运行并完成 Spring 上下文与规则触发流程。

### 7.3 当前测试的不足

当前 `test()` 只要运行过程没有抛异常便会通过，控制台输出不构成自动校验。因此它验证了“流程可执行”，但没有严格验证“积分结果正确”。建议至少增加断言：

```java
import static org.junit.jupiter.api.Assertions.assertEquals;

@Test
void shouldGrant1000ScoreForOrderAtLeast1000() {
    KieSession session = kieContainer.newKieSession();
    try {
        Order order = new Order();
        order.setAmout(1500);

        session.insert(order);
        int firedRules = session.fireAllRules();

        assertEquals(1, firedRules);
        assertEquals(1000, order.getScore());
    } finally {
        session.dispose();
    }
}
```

为覆盖全部金额区间和边界，可采用参数化测试：

```java
import static org.junit.jupiter.api.Assertions.assertEquals;

import java.util.stream.Stream;
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.Arguments;
import org.junit.jupiter.params.provider.MethodSource;

@ParameterizedTest
@MethodSource("orderCases")
void shouldCalculateScoreByAmount(double amount, double expectedScore) {
    KieSession session = kieContainer.newKieSession();
    try {
        Order order = new Order();
        order.setAmout(amount);
        session.insert(order);

        assertEquals(1, session.fireAllRules());
        assertEquals(expectedScore, order.getScore());
    } finally {
        session.dispose();
    }
}

private static Stream<Arguments> orderCases() {
    return Stream.of(
        Arguments.of(99D, 0D),
        Arguments.of(100D, 100D),
        Arguments.of(499.99D, 100D),
        Arguments.of(500D, 500D),
        Arguments.of(999.99D, 500D),
        Arguments.of(1000D, 1000D),
        Arguments.of(1500D, 1000D)
    );
}
```

该测试重点覆盖 100、500、1000 三个区间边界，并同时验证每个订单只激活一条规则。

**注意事项：**

1. `@SpringBootTest` 会启动完整上下文，测试较慢，但能验证 Spring Bean、DRL 资源加载与 Drools 的真实集成。若只验证纯规则逻辑，可构造 KIE 对象做更轻量的单元测试。
2. 由于主类注册了 `CommandLineRunner`，执行 `@SpringBootTest` 时该启动演示也可能随测试上下文执行并输出日志。这会让测试产生额外副作用；生产代码中可用 `@Profile` 或条件化 Bean 将演示逻辑与测试隔离。
3. 建议将当前 `test` 改为表达业务意图的名称，例如 `shouldGrant1000ScoreForOrderAtLeast1000`，便于失败报告定位。

## 8. 常见问题排查

| 现象 | 优先检查项 | 处理建议 |
| --- | --- | --- |
| 启动时报找不到规则文件 | `RULES_CUSTOMER_RULES_DRL` 与资源路径是否一致；资源是否在 `src/main/resources` 下。 | 保持 `rules/order.drl` 与实际目录一致，然后重新执行 Maven 构建。 |
| 规则不触发 | 是否执行了 `session.insert(order)` 和 `session.fireAllRules()`；金额是否落在条件范围内。 | 打印金额与 `fireAllRules()` 返回值，检查 DRL 条件。 |
| DRL 改坏后错误信息不清晰 | 配置类没有检查构建结果。 | 在 `buildAll()` 后检查 `kieBuilder.getResults()` 中的 ERROR 消息。 |
| 一笔订单被重复赋分 | 新规则区间发生重叠，或同一事实重复插入。 | 排查边界、插入次数和规则优先级；用断言校验触发数量。 |
| 服务运行久后内存增长 | 创建会话后未释放。 | 对每个 `KieSession` 使用 `try/finally` 调用 `dispose()`。 |
| 金额边界出现预期外结果 | `double` 精度或比较条件不严密。 | 金额改为 `BigDecimal`，并补充临界值测试。 |

## 9. 当前实现结论

当前项目已完成从 Maven 引入 Drools、加载 DRL、构建 `KieContainer`、插入订单事实、触发规则到输出积分结果的最小闭环。以金额 `1500` 为例，规则 `order_rule_4` 命中，最终积分为 `1000`；现有 Surefire 报告显示 1 个测试通过。

后续优先改进方向是：补充构建错误检查、使用断言和参数化测试覆盖所有边界、把金额字段改为 `BigDecimal`、并将启动演示逻辑从正式订单处理流程中隔离。
